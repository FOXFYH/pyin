文件管理.HTML 与主页面协同关系说明（v3.0 认证权限版）
========================================

一、架构概述
----------------------------------------
主页面（父）内嵌一个 iframe 加载 文件管理.HTML（子）。
两者通过 postMessage 双向通信，各自独立运行，互不直接调用函数。

  主页面（父）
    └── <iframe src="文件管理.HTML">（子）

v3.0 变化：
  - 文件管理器底部固定显示身份栏（设备代号、用户名、密码）
  - 所有云端请求自动附带认证信息
  - 认证失败时红色提示


二、通信机制（postMessage）
----------------------------------------

1. 子→父：文件管理通知主页面
   发送方式：window.parent.postMessage(消息对象, '*')
   接收方式：主页面 window.addEventListener('message', function(e){...})
   消息格式：{ type: '消息类型', ...其他数据 }

2. 父→子：主页面通知文件管理
   发送方式：iframe.contentWindow.postMessage(消息对象, '*')
   接收方式：文件管理 window.addEventListener('message', function(e){...})
   消息格式：{ type: '消息类型', target: 'fileManager', ...其他数据 }

3. 关键注意事项
   - 父→子消息必须带 target: 'fileManager'，否则子页面不处理
   - 子发消息前必须检查 window.parent !== window，否则非iframe环境会报错
   - 父发消息前必须确保 iframe 已加载（frame.contentWindow 存在）
   - 主页面接收消息时，通过 msg.type 判断消息类型，不要用 source 字段过滤
   - 文件管理接收消息时，通过 msg.type 判断消息类型，兼容多种消息格式


三、消息类型清单
----------------------------------------

【子→父】文件管理发给主页面的消息：

  type: 'openFile'          双击打开文件
    携带：{ name, content, id, isOffline? }
    主页面行为：解析content，渲染思维导图，更新当前文件名
    isOffline=true 时表示离线模式打开

  type: 'openDenied'        拒绝打开文件
    携带：{ name, reason, localVer?, cloudVer?, message? }
    主页面行为：提示用户无法打开的原因
    reason 值：'conflict' | 'sync_failed' | 'not_found' | 'error'

  type: 'fileContentUpdated' 文件内容被云端下载更新
    携带：{ name, content }
    主页面行为：如果更新的是当前打开的文件，重新解析渲染

  type: 'fileContentChanged' 文件内容变更通知
    携带：{ name, content }
    主页面行为：更新当前文件内容

  type: 'fileDeleted'       文件被删除
    携带：{ name }
    主页面行为：如果删除的是当前打开的文件，清空编辑区

  type: 'fileRenamed'       文件被重命名
    携带：{ oldName, newName }
    主页面行为：如果重命名的是当前打开的文件，更新引用

  type: 'closeFileManager'  请求关闭文件管理弹窗
    携带：无
    主页面行为：隐藏模态框，恢复同步定时器

  type: 'syncStatusChanged' 同步状态变化
    携带：{ icon }
    主页面行为：更新同步按钮图标

  type: 'syncConflict'      同步冲突
    携带：{ cloudVer, context, fileName? }
    主页面行为：提示用户处理冲突

  type: 'conflictResolved'  冲突已解决
    携带：无
    主页面行为：恢复正常状态

  type: 'syncToast'         同步状态通知
    携带：{ message }
    主页面行为：显示同步提示

  type: 'syncSettingsChanged' 同步设置变化
    携带：config对象
    主页面行为：预留，当前无操作

  type: 'registerResult'    注册新文件结果
    携带：{ success, name?, id?, message? }
    主页面行为：根据结果处理

  type: 'importResult'      导入文件结果
    携带：{ success, name?, id?, action?, message? }
    主页面行为：根据结果处理

  type: 'fileSelected'      单击选中文件
    携带：{ name, content, id }
    主页面行为：预留，当前无操作


【父→子】主页面发给文件管理的消息：

  type: 'initConfig'        初始化配置
    携带：{ appPrefix?, config? }
    文件管理行为：更新APP_PREFIX和同步配置，刷新列表

  type: 'open'              打开文件管理器
    携带：{ currentFileName, currentContent }
    文件管理行为：初始化界面，渲染文件列表，刷新云端列表

  type: 'setCurrentFile'    切换当前文件
    携带：{ currentFileName, currentContent }
    文件管理行为：更新文件名输入框，刷新列表

  type: 'openFile'          请求打开文件（走先一致再打开策略）
    携带：{ id? | name? }
    文件管理行为：执行一致性检查后回复 openFile 或 openDenied

  type: 'requestOpenFile'   请求打开文件（同openFile）
    携带：{ id? | name? }
    文件管理行为：同上

  type: 'registerNewFile'   注册新创建的文件
    携带：{ name, content, folder? }
    文件管理行为：创建文件，自动上传云端，回复 registerResult + openFile

  type: 'importFileData'    导入文件数据
    携带：{ name, content, folder? }
    文件管理行为：强制新ID创建文件（不覆盖已有），回复 importResult + openFile

  type: 'contentChanged'    当前文件内容已变更
    携带：无
    文件管理行为：同步本地数据到管理器

  type: 'uploadCurrent'     请求上传当前文件
    携带：无
    文件管理行为：同步本地数据后上传

  type: 'syncCurrentFromMain' 主页面触发同步当前文件
    携带：无
    文件管理行为：同步本地数据后执行同步

  type: 'syncAllFiles'      请求同步所有文件
    携带：无
    文件管理行为：执行全部同步

  type: 'downloadCloudFile' 下载指定云端文件
    携带：{ fileId, fileName? }
    文件管理行为：下载指定文件

  type: 'refreshCloud'      刷新云端文件列表
    携带：无
    文件管理行为：重新读取云端索引

  type: 'refreshFileList'   刷新文件列表显示
    携带：无
    文件管理行为：重新渲染文件列表

  type: 'fileDeleted'       通知：当前文件已删除
    携带：{ name }
    文件管理行为：清空当前文件引用，刷新列表

  type: 'fileRenamed'       通知：当前文件已重命名
    携带：{ oldName, newName }
    文件管理行为：更新当前文件名，刷新列表

  type: 'fileDeleted_local' 通知：应用层已删除文件（同步删云端）
    携带：{ fileId }
    文件管理行为：删除云端对应行及拆分片段

  type: 'fileRenamed_local' 通知：应用层已重命名文件（同步改云端）
    携带：{ fileId }
    文件管理行为：更新云端文件名（含拆分片段）

  type: 'fileCopied'        通知：应用层已复制文件（同步上传云端）
    携带：{ fileId }
    文件管理行为：上传文件副本到云端

  type: 'syncComplete'      同步完成通知
    携带：{ pulled, uploaded }
    文件管理行为：显示同步结果

  type: 'mainReady'         主页面就绪
    携带：无
    文件管理行为：启动自动同步

  type: 'mainActivated'     主页面激活
    携带：无
    文件管理行为：启动自动同步


四、职责划分
----------------------------------------

主页面负责：
  - 编辑当前打开文件的内容
  - 自动保存内容到 localStorage
  - 保存视图状态（缩放/平移）
  - 内容变更后递增版本号
  - 自动云同步（定时上传）

文件管理负责：
  - 文件列表展示（本地+云端，含字符数、版本号、同步状态）
  - 新建/重命名/删除/复制文件
  - 导入（.json文件）/导出（.json文件，内容自动美化）
  - 文件夹管理（新建/导航/删除）
  - 手动云同步（同步当前/全部同步/刷新云端）
  - 同步设置配置
  - 身份认证信息管理（设备代号/用户名/密码，底部固定栏）

传到云端的数据字段（v3.0）：
  表格名    — 数据表名
  操作      — 操作类型
  用户名    — 认证用户名
  密码      — 认证密码
  锁登记处  — 设备标识（写操作自动附带）
  ID        — 文件唯一标识，创建后不可改
  文件名    — 只有文件管理能改
  内容      — 主页面编辑时改，文件管理同步下载时改
  版本号    — 主页面编辑后递增，文件管理同步时更新
  最后上传发起时间 — 上传时自动生成
  拥有者    — 写入时自动填入当前用户（如果为空）


五、身份认证栏（v3.0新增）
----------------------------------------

文件管理器底部固定显示身份栏（position:fixed;bottom:0），包含：

  - 设备代号：只读，随机生成后持久化（localStorage: mindmap_device_id）
    格式：D + 时间戳(36进制) + 4位随机字符
    用途：云端抢锁标识

  - 用户名：可编辑，默认 test（localStorage: mindmap_auth_user）

  - 密码：明文可编辑，默认 1234（localStorage: mindmap_auth_pass）

  - 保存按钮：将当前用户名密码写入 localStorage

  - 状态提示：保存成功/认证失败

认证失败时：
  - 云端返回 { 需要认证: true }
  - 身份栏状态文字变红显示"认证失败"
  - toast 提示具体错误信息


六、复用文件管理.HTML 的接入指南
----------------------------------------

1. 在主页面中添加模态框和iframe：
   <div id="file-modal" onclick="if(event.target===this)closeFileManager()">
     <iframe id="fileManagerFrame" src="文件管理.HTML"></iframe>
   </div>

2. 打开文件管理器时，发送初始化消息：
   var frame = document.getElementById('fileManagerFrame');
   frame.contentWindow.postMessage({
     target: 'fileManager',
     type: 'open',
     currentFileName: '当前文件名',
     currentContent: '当前内容'
   }, '*');

3. 注册消息监听器，处理文件管理发来的消息：
   window.addEventListener('message', function(e) {
     if (!e.data || typeof e.data !== 'object') return;
     var msg = e.data;
     if (!msg.type) return;
     switch (msg.type) {
       case 'openFile':
         // 用 msg.name 和 msg.content 打开文件
         break;
       case 'openDenied':
         // 提示无法打开：msg.reason
         break;
       case 'fileContentUpdated':
         // 更新文件内容：msg.name, msg.content
         break;
       case 'fileDeleted':
         // 清空编辑区（如果是当前文件）
         break;
       case 'closeFileManager':
         // 隐藏模态框
         break;
       case 'syncStatusChanged':
         // 更新同步按钮图标：msg.icon
         break;
       case 'syncConflict':
         // 处理冲突：msg.cloudVer, msg.context, msg.fileName
         break;
       case 'registerResult':
         // 注册结果：msg.success, msg.name, msg.id
         break;
       case 'importResult':
         // 导入结果：msg.success, msg.name, msg.id
         break;
       // ... 其他消息类型按需处理
     }
   });

4. 必须注意的坑：
   - 消息监听器必须在所有代码之前注册，不能放在可能报错的代码后面
   - 主页面中如果引用了文件管理才有的DOM元素（如fileList），必须加空值保护
   - 父→子消息必须带 target: 'fileManager'，否则子页面忽略
   - 子页面发消息前必须检查 window.parent !== window
   - iframe 的 src 路径要正确，跨域时 postMessage 的 origin 参数不能为 '*'
   - v3.0 所有云端操作需要认证，确保"账号管理"表已创建


七、踩坑记录
----------------------------------------

1. DOM元素不存在导致TypeError
   拆分版把文件管理拆到iframe后，主页面不再有 id="fileList" 等元素，
   但代码中 document.getElementById('fileList').addEventListener(...) 仍然存在，
   返回null后报TypeError，导致后续代码（包括消息监听器注册）全部不执行。
   修复：所有引用可能不存在的DOM元素，必须加空值保护。

2. 两个消息监听器冲突
   拆分版有两个 window.addEventListener('message',...) ，
   一个用 source 过滤，一个不过滤，导致消息被重复处理或被拦截。
   修复：合并为一个消息监听器，统一用 type 判断。

3. source 字段过滤不可靠
   子页面用 {source:'fileManager', type:'xxx'} 发消息，
   父页面用 e.data.source!=='fileManager' 过滤，
   但展开运算符 ...data 可能导致字段覆盖或丢失。
   修复：去掉 source 过滤，改用 Object.assign 构建消息，直接用 type 判断。

4. 父→子消息未带target被忽略
   父页面发消息未带 target:'fileManager'，子页面监听器直接 return，
   导致初始化消息丢失。
   修复：所有父→子消息必须带 target: 'fileManager'。

5. 文件索引重复
   多处 idx.push 未检查同名文件是否已存在，导致同一文件出现两条记录。
   修复：setFileIndex 自动按文件名去重，同名只保留最后一条。

6. 字符数显示0
   本地文件缓存被清理后 loadFileDataById 返回 null，字符数显示0。
   修复：优先用本地数据，没有则用 contentLength 字段；云端文件显示云端字符数。

7. 认证失败无提示
   v3.0 新增认证后，认证失败时客户端无反馈。
   修复：检测云端返回的"需要认证"字段，红色提示并toast。
